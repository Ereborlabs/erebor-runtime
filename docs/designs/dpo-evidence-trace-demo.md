## Problem Statement

Erebor is close to a concrete OpenClaw pilot: one governed session, one agent actor, browser/process mediation, policy decisions, and a shared audit trail. The prior wedge framed the buyer as a security engineering or AI-platform lead who wants to allow a risky agent workflow without trusting the agent.

The revised hypothesis is sharper: the first urgent approval stakeholder may be a Data Protection Officer, privacy lead, or GRC owner who is asked to approve agents that may touch personal data or PII. Security engineering or AI platform may still own deployment and budget.

Without an Erebor-controlled execution path, that approval stakeholder cannot answer the basic review questions:

- Which governed pages, resources, tools, and process surfaces were exposed to the agent?
- Which systems did the agent touch?
- Did it attempt to grant new authority, exfiltrate data, launch a side-channel browser, or run a risky process?
- Which controls blocked or held the action?
- What residual risk remains outside the governed surfaces?

The product claim should not be “Erebor makes you GDPR/HIPAA compliant.” That is too broad and legally sloppy. The stronger claim is: Erebor produces defensible runtime evidence and enforcement for governed agent sessions, so a DPO or compliance owner can review a specific agent workflow without relying on the agent’s own promises.

For the first discovery sprint, the primary cohort should be EU-facing B2B SaaS privacy/DPO/GRC stakeholders. Healthcare/HIPAA/ePHI is a later variant unless a warm healthcare lead appears immediately.

## Demand Evidence

Current demand evidence is still hypothesis-level, not buyer proof.

What supports the hypothesis:

- The repo’s OpenClaw pilot now proves the right technical shape: governed browser mediation, process interception, deterministic policy denial, and JSONL audit direction under one session boundary.
- The ICO’s AI/data-protection guidance frames AI governance around accountability, DPIAs, DPO involvement, data minimisation, risk assessment, and demonstrating mitigations.
- HHS’s HIPAA Security Rule summary emphasizes access control, audit controls, risk analysis, documentation, and safeguards around ePHI. This supports the later healthcare variant, not the first discovery lane.
- The user’s instinct is concrete: “without Erebor it is almost impossible to run agent, and make sure if agent read the PIIs or not, what it did and so on.” That is closer to buyer language than the older “universal action-governance runtime” abstraction.

What is missing:

- No named DPO or privacy lead has confirmed this is a top-three blocker.
- No buyer has said an Erebor evidence trace would change approval from “blocked” to “pilot allowed.”
- No organization has described the exact audit fields, DPIA inputs, retention rules, or residual-risk language they would need.
- No compliance owner has said they would pay for this, sponsor it, or introduce the security/AI-platform owner who controls deployment.

Position: this is a better buyer hypothesis than generic “enterprise AI governance,” but it is not demand yet.

## Status Quo

The current status quo is unknown, but the likely patterns are:

- Block the agent workflow entirely when it may touch production personal data.
- Allow only synthetic-data demos or isolated lab environments.
- Require manual review, screenshots, policy checklists, DPIAs, and security questionnaires before any real deployment.
- Push the problem to security engineering, who then asks for logs, access controls, network restrictions, endpoint controls, and evidence of least privilege.
- Treat the agent as an unbounded processing risk when personal data is involved.

The real competitor is probably not another startup. It is “no, you cannot run that agent on real data” plus a spreadsheet/process workaround.

## Target User & Narrowest Wedge

First discovery lane:

- EU-facing B2B SaaS company, 200-2000 employees.
- Role: DPO, privacy lead, GRC lead, or compliance-backed security lead.
- Adjacent champion: security engineering manager or AI-platform lead who owns the technical deployment path.
- Their job: allow useful AI adoption without creating a data-protection incident, regulatory finding, or unprovable processing activity.
- They get promoted for enabling safe AI adoption with evidence.
- They get blamed when an agent sees data it should not, creates uncontrolled authority, or leaves no defensible audit trail.

Narrowest wedge:

Package the OpenClaw pilot as a DPO-ready evidence trace for one governed support-investigation workflow: run the agent, block the risky authority transition, then render the reviewable packet from the governed session.

The first milestone can be a manually rendered markdown evidence trace generated from one real governed run and its JSONL audit. Automation comes after buyers confirm the trace fields and review shape.

### How To Read The Trace

- Governed session: one Erebor-owned run with one actor, one policy set, and correlated surfaces.
- Controlled surface: a browser, process, tool, API, or filesystem path whose effects pass through Erebor before execution.
- Exposed resource: a URL, target, command, tool, file path, or API endpoint made available to the agent through a controlled surface. This is not the same as proving every personal-data value the agent semantically read.
- Authority transition: an action that creates, expands, or bypasses access, such as an OAuth grant, unmanaged browser endpoint, credential access, write action, deployment, or external data transfer.
- Held action: an action that does not execute until approval exists.
- Denied action: an action rejected before it reaches the browser, process, or tool.
- Enforced control: Erebor or its runner blocked or mediated the effect before execution.
- Cooperative control: configuration steered the agent into Erebor, but a bypass may remain outside the governed path.
- Observed control: Erebor recorded or reported the behavior, but did not prevent it.
- Deferred control: a risk Erebor does not claim to enforce in this pilot.
- Defensible evidence: records that can support internal review. It is not legal advice and not a compliance guarantee.

### One-Page Evidence Trace Outline

1. Executive summary: purpose, actor, date, result.
2. Scope: controlled surfaces, policy version, non-claims.
3. Session timeline: allowed actions and denied/held actions.
4. Authority transitions: OAuth grants, raw-CDP attempts, risky process launches.
5. Data exposure statement: governed resources exposed; content classification status.
6. Residual risk: enforced, cooperative, observed, deferred controls.
7. Attachments: JSONL audit, policy hash, config hash, runbook, reviewer notes.

## Constraints

- Do not claim GDPR compliance, HIPAA compliance, certification, or legal sufficiency.
- Do not claim Erebor proves the agent never read PII outside governed surfaces.
- In v1, guarantee URL/action/resource provenance, not semantic PII classification.
- Do not claim Linux host pilot equals whole-device containment.
- Do not claim OpenClaw integration is the enforcement boundary. Enforcement is the Erebor-controlled execution path.
- Do not sell generic audit after the fact. The point is pre-action control plus audit evidence.
- Do not make DPO the only buyer. DPO/privacy may own approval; security/AI platform may own deployment and budget.
- The first pack must be self-hosted and reproducible because the trust boundary matters.

## Premises

1. The DPO wedge is promising, but right now it is unvalidated demand. No named DPO has pulled yet.
2. The sharper problem is not “can we get GDPR/HIPAA certified?” It is “can we produce defensible evidence of what the agent did around personal-data workflows?”
3. Erebor’s honest claim should be “governed session evidence and enforcement for controlled surfaces,” not “we guarantee the agent never saw PII.”
4. The OpenClaw OAuth pilot still fits, but the buyer takeaway should become a DPIA/accountability packet: purpose, resources exposed, authority attempted, policy decision, residual risk.
5. The first sales motion should test DPO/privacy plus security/GRC together, because DPO may own approval while security/AI platform may own deployment and budget.

## Landscape Notes

The conventional compliance framing is documentation-first: DPIAs, records of processing, policy reviews, risk registers, and audit logs. That is useful, but it becomes weak when the system is an autonomous agent whose runtime behavior changes with prompt, page content, tool output, and process access.

Erebor’s contrarian angle is that agent compliance evidence should be generated at the execution boundary while the agent runs. The DPO does not need another policy PDF. They need a bounded workflow where the runtime can say: this was the actor, this was the purpose, these were the pages/processes/tools touched, these actions were allowed, these authority transitions were denied, and this residual risk remains.

Sources reviewed:

- ICO, “Guidance on AI and data protection,” especially accountability, governance, DPIA, DPO, risk, and mitigation sections: https://ico.org.uk/for-organisations/uk-gdpr-guidance-and-resources/artificial-intelligence/guidance-on-ai-and-data-protection/
- HHS, “Summary of the HIPAA Security Rule,” especially risk analysis, access control, audit controls, and documentation requirements: https://www.hhs.gov/hipaa/for-professionals/security/laws-regulations/index.html
- European Commission archive links to Article 29/EDPB DPO and DPIA guidance: https://ec.europa.eu/newsroom/article29/items/612048/en?doc_id=44100 and https://ec.europa.eu/newsroom/article29/items/611236/en?item_id=611236

## Approaches Considered

### Approach A: Evidence Trace Demo

Summary: Package the OpenClaw pilot as a live evidence trace: the agent runs, Erebor allows safe investigation, blocks a high-risk authority transition, then immediately renders the session into a DPO-readable evidence trace report. This is smaller than a full live DPIA product, but much stronger than a static report because the buyer sees evidence being created from the controlled execution path.

Effort: M/L
Risk: Medium

Pros:

- Creates a real whoa moment without building the full compliance runtime.
- Keeps the almost-finished OpenClaw pilot as the core proof.
- Shows the DPO value in the moment: action, block, evidence, residual risk.
- Still lets interviews start now with a sketch before the demo is polished.

Cons:

- Requires browser provenance and OAuth denial records to be credible.
- Needs a small report renderer or a fast manual rendering workflow.
- Must avoid pretending it performs semantic PII classification or legal review.

Reuses: governed OpenClaw pilot, OAuth denial demo, session-mediated browser config, process interception audit, planned shared JSONL audit, prior office-hours design.

The whoa moment:

1. OpenClaw investigates a realistic support/OAuth thread through Erebor.
2. The agent reaches a high-risk OAuth authority transition or raw-CDP bypass attempt.
3. Erebor blocks it before execution.
4. The screen immediately shows: purpose, actor, governed resources exposed, allowed actions, blocked authority transition, policy rule, and residual risk.
5. The buyer can say: “This is the packet I would attach to an internal AI/privacy approval review.”

Minimum viable demo:

- One real governed OpenClaw run using the OAuth lab or deterministic equivalent.
- Browser navigation/resource provenance records and the OAuth denial record are required for the DPO evidence-trace claim.
- One JSONL audit stream for process records is required for the process/raw-CDP authority-transition claim.
- One generated or manually rendered markdown evidence trace report using the schema above.
- One explicit residual-risk section that labels host containment limits and missing content classification.
- If browser provenance or OAuth denial records are unavailable, downgrade the demo claim to “process and raw-CDP authority-transition evidence only” and do not pitch it as a DPO-ready personal-data workflow packet.


### Approach B: Compliance Runtime

Summary: Build the ideal long-term product architecture: Erebor feeds DPIA, ROPA, audit, approvals, data-flow inventory, policy decisions, retention, and residual-risk reports across browser, terminal, MCP, SaaS, filesystem, and API surfaces.

Effort: XL
Risk: High

Pros:

- Strongest long-term enterprise product.
- Gives DPO/security/legal a system of record rather than a one-off artifact.
- Matches the universal governed-session thesis.

Cons:

- Too much to build before validating demand.
- Easy to become a governance dashboard instead of an enforcement runtime.
- Requires more surfaces, integrations, and policy semantics than the first buyer test needs.

Reuses: session-hypervisor plan, semantic classification plan, audit model, future policy/approval surfaces.

### Approach C: Live DPIA Demo

Summary: Turn the demo into theater for the DPO: as OpenClaw runs, Erebor generates a live DPIA-style trace showing purpose, data categories, decisions, risk, mitigations, and the final residual-risk summary.

Effort: L
Risk: Medium

Pros:

- Memorable and highly differentiated.
- Makes the compliance value obvious in a 10-minute call.
- Could create a “whoa” moment for privacy teams used to static documents.

Cons:

- Requires UI/report polish and possibly data-classification work.
- Risks looking like compliance automation magic before the enforcement story is trusted.
- Could distract from the hard boundary proof if built too early.

Reuses: incident demo, OAuth lab, JSONL audit, future audit-tail/report rendering.

## Recommended Approach

Choose Approach A revised: Evidence Trace Demo.

Do not wait to interview. Start interviewing now with the evidence-trace sketch, then use the demo as a closing artifact for people who lean forward. The first learning goal is not “can we impress them?” It is “does this packet match the evidence they need to approve an agent near personal data?”

This is better than the original static Evidence Pack because it has a product moment: the agent acts, Erebor blocks the risky transition, and the compliance-readable packet appears from the run. It is smaller than the full Live DPIA Demo because v1 does not need automatic data-category classification, a polished dashboard, ROPA automation, retention workflows, or legal signoff routing.

The positioning sentence to test:

> Erebor lets you run an AI agent inside a governed session, blocks risky authority transitions before they execute, and turns the run into a DPO-readable evidence trace: what governed resources were exposed, what the agent attempted, what was blocked, and what residual risk remains.

Sharper version for outreach:

> If your team wants agents to work near PII, the blocker is not only model safety. It is proof. Erebor creates a governed session and an evidence trace your DPO can actually review.


## Open Questions

Must answer before productizing:

1. Is the DPO/privacy lead the buyer, the blocker, the evaluator, or only a required stakeholder?
2. Which evidence-trace field would actually change approval: resources exposed, denied authority transitions, policy version, residual risk, or reviewer signoff?

Important but secondary:

- Does the DPO want runtime evidence directly, or do they delegate all technical review to security/GRC?
- Which phrase resonates: “DPIA evidence,” “agent audit packet,” “AI processing record,” “governed session report,” or “governed resource exposure trace”?
- Does healthcare care enough about agentic workflows now, or is EU-facing SaaS a faster first wedge than HIPAA-heavy healthcare?
- Does semantic “agent read PII” proof matter in the first demo, or is surface/action provenance enough for the first buyer call?

## Success Criteria

Discovery success criteria:

- Start interviews immediately, before the demo is fully polished.
- Contact 30-40 DPO, privacy, GRC, security engineering, or AI-platform leads, prioritizing EU-facing B2B SaaS.
- Get 5 conversations. If 5 buyer conversations do not materialize, get at least 2 expert calls with privacy counsel/DPO advisors and identify why buyer access failed.
- At least 3 can name a real agent workflow currently blocked or forced into synthetic data because of privacy/compliance risk.
- At least 2 react strongly to the live evidence-trace concept, not just the general compliance story.
- At least 1 agrees to review the OpenClaw evidence trace against their internal AI/privacy approval checklist.

Demo success criteria:

- The OpenClaw pilot completes the governed support/OAuth workflow or a deterministic equivalent.
- The governed run shows allowed investigation actions and denied authority transitions.
- The audit stream uses one session id across browser and process surfaces where implemented.
- The evidence trace appears during or immediately after the run, even if v1 is a generated markdown report rather than a dashboard.
- A non-engineering compliance stakeholder can understand the trace in under 10 minutes.
- The trace labels enforced, cooperative, observed, and deferred controls clearly.
- The trace includes residual risk instead of hiding it.


## Distribution Plan

Initial distribution remains self-hosted.

- Ship the DPO evidence trace as part of the governed OpenClaw pilot package, not as a separate product.
- The package should include policy files, session config, a runbook, generated JSONL audit, and a rendered evidence-trace markdown or HTML report.
- For external design partners, distribute through a GitHub release or private repo bundle with exact verification commands.
- Do not build a hosted compliance dashboard until a buyer asks for repeated review, retention, search, or team workflows.
- If healthcare/HIPAA becomes the wedge, package deployment guidance around customer-controlled infrastructure and do not process ePHI through a remote Erebor service by default.

## Dependencies

- Live OAuth validation with throwaway GitHub data, or a deterministic equivalent if live validation slips.
- Browser CDP JSONL audit output wired into the configured shared sink. If this is unavailable, the DPO evidence-pack claim must be downgraded to process/raw-CDP authority-transition evidence only.
- Final call-pressure demo script updated from “blocked OAuth” to “DPO evidence trace.”
- A manual report template that renders the session audit into compliance-readable language.
- Careful claim language around PII/ePHI: surface/action evidence first, content classification later.
- Buyer discovery before expanding into full compliance-runtime architecture.

## The Assignment

Start interviewing now.

Do not wait for the demo to be perfect. Use the current evidence-pack schema and the whoa-moment storyboard as the conversation artifact. The point of the first calls is to discover whether the live trace is the thing DPO/privacy/GRC people actually want, and which fields would make it reviewable.

Primary target list:

- DPOs and privacy leads at EU-facing B2B SaaS companies.
- GRC leads who own AI governance review.
- Security engineering managers supporting privacy reviews.
- AI-platform leads trying to get internal agent workflows approved.

Outreach channels:

- Warm intros first.
- LinkedIn DMs second.
- Privacy/GRC Slack or community groups if available.
- Founder/network asks: “Who reviews AI-agent access to customer data at your company?”

Day plan:

- Day 0: Send 5 warm messages today with the evidence-trace framing. Do not wait for a polished demo.
- Day 1: Build a list of 40 names and sort them into DPO/privacy, GRC, security, and AI-platform.
- Day 2: Send 15 warm or semi-warm messages.
- Day 3: Send 15-25 cold LinkedIn/email messages and ask for intros from any replies.
- Day 4: Run first calls or, if no calls, talk to 2 privacy counsel/DPO advisors to sharpen language.
- Day 5: Show the evidence-trace storyboard and schema to anyone willing to react.
- Day 6: Revise the trace fields and whoa moment based on objections.
- Day 7: Decide: continue DPO-first, make DPO a stakeholder with security/AI-platform as buyer, or pivot the demo back to security enforcement.

Send this message:

> We are building a self-hosted runtime that lets an AI agent run inside a governed session. The demo moment is: the agent investigates a realistic workflow, hits a risky authority transition, Erebor blocks it, and the run turns into an evidence trace for privacy/compliance review: what governed resources were exposed, what the agent attempted, what was blocked, and what residual risk remains. I am trying to learn whether this would change how DPOs approve agents near PII. Could I show you the 2-minute storyboard and get your blunt reaction?

Ask these questions:

1. When a team wants to use an AI agent near personal data, who says yes or no?
2. What is the current review workflow?
3. What evidence do you need that current tools do not produce?
4. Would a governed-session evidence trace change the answer, or is the workflow still blocked?
5. Which field in the trace would make or break approval?
6. Would the live moment matter, or is a static report enough?
7. Who owns budget if this matters: DPO, GRC, security, AI platform, legal, or the product team?

Exit criteria:

- Continue DPO wedge if 2+ target buyers ask to review the trace or introduce the technical owner.
- Build the demo if at least 2 people say the live trace would materially change the conversation compared with a static audit packet.
- Keep DPO as stakeholder, not buyer, if privacy likes it but sends you to security/AI-platform for budget and deployment.
- Drop DPO-first positioning if calls say existing GRC/DPIA workflows are not the blocker and the real blocker is endpoint/security enforcement.

The goal is not compliments. The goal is one person who says: “If you can produce that trace from a real run, I will compare it to our internal approval checklist.”
