## Problem Statement

Erebor is trying to prove that a governed execution session can let AI agents operate near real systems without trusting the agent to police itself.

After the latest interview synthesis, the strongest product wedge remains:

> Erebor is not the agent. It is the control boundary and evidence trace around the agent.

Filesystem revert is a candidate differentiator because it makes reversal and containment tangible. A deployment owner can see a protected directory enter the session, watch the agent mutate a reversible view, inspect the checkpoint diff, and then either promote or roll back the result.

The danger: "filesystem revert" can easily become a storage-system project. The startup question is narrower: does a reversible filesystem session help an agent vendor or deployment owner move a workflow from "blocked near real state" to "approved under constraints"?

## Demand Evidence

Current evidence level: reviewer concern, not buyer demand.

Signals from prior interviews and design docs:

- Tony's minimum evidence was an audit trail that explains what happened and supports reversal or containment.
- George's strongest phrase was "actual blockade."
- Alex wanted step-by-step evidence, blocked attempts, approval before sensitive or irreversible actions, logs outside the agent's control, and a stop/revoke story.
- Prior designs still note that no DPO/GRC interview has produced a named deployment owner with a blocked workflow.

Verdict: revert is not yet validated as a buying reason. It is validated as a plausible proof requirement for the governed-session wedge.

## Status Quo

The working status quo is avoidance: teams keep agents away from real files, config, local secrets, credentials, or mutable state.

That means workflows either stay manual, run only in synthetic sandboxes, or require humans to approve and clean up risky changes. The cost to test is whether a reversible session changes that posture from "do not let the agent touch it" to "let the agent operate, then inspect and commit."

## Target User & Narrowest Wedge

Target human: agent vendor or deployment owner.

They need to answer a reviewer or customer question:

> If this agent touches real local state, how do we know what changed, who approved it, and whether we can recover?

Narrowest wedge:

- one governed session
- one protected non-code directory, such as local app config or agent workspace state
- one agent-caused mutation to the reversible view
- one checkpoint showing before, after, and diff
- one promotion gate
- one rollback proof
- one evidence packet that names residual risk

This should not start with broad home-directory claims, arbitrary per-action revert, single-file edge cases, privileged metadata completeness, or cross-platform endpoint guarantees.

## External Landscape

Current research supports the direction but weakens the claim that rollback alone is unique.

- Cordon argues that per-tool-call guardrails are the wrong boundary for multi-step agents; it proposes task-level semantic transactions with commit, rollback, recovery, and audit over staged effects and reversible state: https://arxiv.org/abs/2606.17573
- Crab argues that agent sandbox recovery needs host-side checkpoint/restore informed by OS-visible effects, because chat-only recovery misses filesystem/process/runtime state: https://arxiv.org/abs/2604.28138
- ClawGuard validates the adjacent task-boundary story: deterministic runtime enforcement against adversarial tool calls can protect tool-using agents without relying on model alignment alone: https://arxiv.org/abs/2604.11790

Implication: Erebor should not pitch "we invented rollback for agents." The better claim is:

> Erebor turns reversal and containment into a governed-run evidence packet for real agent deployment decisions.

## Premises

1. Filesystem revert should be framed as reversal/containment proof inside a governed execution session, not as generic file backup or developer undo.
2. The first human is an agent vendor or deployment owner trying to move a workflow from "blocked near real state" to "approved inside a reversible session."
3. The first filesystem surface must prove a real before/after/rollback story for protected non-code state, while honestly labeling browser/process/network surfaces it does not yet control.
4. The market already understands guardrails, sandboxes, audit, and increasingly rollback; Erebor wins only if it turns those into one credible governed-run evidence packet.

## Approaches Considered

### Approach A: Narrow Revert Evidence Demo

Summary: Build or storyboard one protected directory volume in a governed session: base snapshot, agent-mutated overlay, checkpoint diff, evidence packet, promotion, and rollback.

Effort: M
Risk: Medium

Pros:

- Tests the buyer-path claim quickly.
- Reuses the OSTree/OverlayFS v2 plan.
- Creates a visceral before/after/rollback artifact.

Cons:

- Not full filesystem coverage.
- Must label limits clearly.
- Requires enough Linux/Docker work to be credible.

Reuses: governed session model, session audit, OpenClaw pilot evidence packet, OSTree/OverlayFS v2 plan.

### Approach B: Full Filesystem Revert Surface

Summary: Implement the v2 plan as a first-class surface: config roots, per-session OSTree repo, overlay mounts, layer normalization, result commits, promotion, rollback, metadata sidecars, and audit records.

Effort: XL
Risk: High

Pros:

- Strongest technical claim.
- Becomes a real platform primitive.
- Supports checkpoint, session, and promotion revert cleanly.

Cons:

- Too much before buyer validation.
- High kernel and metadata edge-case surface.
- Risks becoming a storage project instead of a sales wedge.

Reuses: session interception refactor, filesystem operation kinds, session registry, OSTree/OverlayFS v2 plan.

### Approach C: Evidence Packet First, Runtime Later

Summary: Create a buyer-facing mock or semi-manual evidence packet that shows mandate, file changes, rollback proof, residual risk, and reviewer questions without committing to full runtime implementation yet.

Effort: S
Risk: Medium

Pros:

- Fastest interview learning.
- Can be shown to agent vendors immediately.
- Avoids overbuilding if buyers do not care.

Cons:

- Weaker actual-blockade moment.
- Can be dismissed as a slide/demo artifact.
- Does not prove hard containment.

Reuses: prior DPO evidence-trace design, existing audit narrative, manual verification commands from the filesystem plan.

## Recommended Approach

Choose Approach A: Narrow Revert Evidence Demo.

It is the right balance for the current evidence level. Reviewer concern is strong enough to build a credible demo, but not strong enough to justify the full filesystem surface before the buyer-path test.

The first demo should show:

1. A protected non-code directory imported as the base state.
2. The agent sees only the reversible session view, not the original host path.
3. The agent mutates the session view.
4. Erebor creates a checkpoint with changed files and deletions represented in an inspectable manifest.
5. The evidence packet shows base, result, diff, decision, and residual risk.
6. Promotion is explicit.
7. Rollback restores the pre-promotion state.

The product sentence:

> Erebor lets an agent operate near protected local state in a governed session, then gives the deployment owner the proof to inspect, promote, or roll back the changes.

## Open Questions

- Which workflow gives the demo a believable buyer context: agent vendor install/config, local credentials-adjacent config, generated customer-support artifacts, or internal ops state?
- Is the first demo allowed to be Docker/OCI-only, or must it work on Linux host bare metal to feel real to deployment owners?
- What exact reviewer sentence changes after seeing the rollback artifact?
- Which residual risks must be named in the first packet so Erebor does not overclaim containment?

## Success Criteria

The idea gets stronger if the next five agent-vendor or deployment-owner conversations produce:

- 2 people who say reversible filesystem session evidence would help move a blocked or narrowed workflow forward.
- 1 person who names a real workflow where agents are kept away from real files, config, or local state.
- 1 person who asks for the artifact in their own review process.
- 1 person who distinguishes this from tool-call guardrails or enterprise monitoring.

The idea gets weaker if:

- They say tool-call validation is enough.
- They say disposable sandboxes are enough.
- They do not care about promotion/rollback proof.
- They cannot name a workflow blocked near real mutable state.

## Distribution Plan

For the proof stage, distribution is not a general binary release.

Ship the demo as:

- a reproducible runbook in the repo
- a generated evidence packet artifact
- a short demo script for interviews
- a capability/residual-risk table

Do not package this as a product feature until a design partner or deployment owner says the artifact changes a real review decision.

## Dependencies

- The filesystem OSTree/OverlayFS v2 plan remains the technical source of truth.
- The governed session audit path must be able to include filesystem checkpoint, promotion, and rollback records.
- The demo must not claim whole-device containment.
- The demo should prefer directory volumes first and explicitly defer single-file bind mounts unless required for the buyer proof.

## The Assignment

In the next three buyer-path conversations, do not ask "is revert useful?"

Show or describe the narrow revert evidence packet and ask:

1. "Do you currently block agents from touching real config, local state, credentials-adjacent files, or generated operational artifacts?"
2. "If yes, who owns that decision?"
3. "Would a governed run with before/after diff, promotion gate, rollback proof, and residual-risk statement change the decision?"
4. "Would this be enough, or would you still require full sandbox isolation, endpoint control, or enterprise monitoring?"
5. "What exact workflow would you let through that you would not let through today?"

Classify the result:

- Strong signal: names a blocked workflow and says the packet would change approval.
- Medium signal: likes rollback but still needs broader sandbox/process/browser proof.
- Weak signal: says it is nice but cannot name the workflow or owner.
- Negative signal: says existing sandbox/tool guardrails already solve it.

## What I Noticed About How You Think

- You noticed "revert" as a differentiator after interviews, not from architecture-first wandering. That is good. Keep tying it to interview language.
- Your instinct is strongest when you translate technical control into reviewer proof: reversal, containment, promotion, rollback, residual risk.
- The risk is overbuilding the filesystem layer before someone says it changes approval. Treat the first implementation as a buyer-learning instrument.
- The phrase to keep close is not "filesystem revert." It is "inspect, promote, or roll back an agent's changes before they become real."
