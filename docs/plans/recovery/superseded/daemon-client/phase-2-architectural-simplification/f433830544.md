## Purpose

Simplify the implemented Phase 2 architecture before later phases add
packages, adapters, more runners, PTY support, and additional governed
surfaces.

This document is deliberately a review proposal, not implementation
authorization. Each suggestion describes:

- what the code does now;
- why the current ownership is difficult to extend or reason about;
- a simpler target;
- exact current and proposed code ownership;
- behavior that must remain unchanged;
- compatibility or product-contract changes;
- risks and acceptance evidence; and
- an independent approve, reject, or revise decision.

The user may approve suggestions independently. Approval of one suggestion
does not approve another suggestion or authorize starting a later phase.

## How To Review This Proposal

For each suggestion, select exactly one decision:

- `[ ] Approve`
- `[ ] Reject`
- `[ ] Revise`

When selecting `Revise`, place the replacement decision under `Review notes`.
Implementation must not begin while any required decision for that suggestion
is unresolved.

Suggestions marked behavior-preserving reorganize ownership without changing
the daemon protocol or session result. Suggestions marked contract-changing
remove or alter currently exposed Phase 2 fields or messages and require an
explicit decision.

## Non-Negotiables

These are not simplification candidates:

- `erebord` continues to host daemon control and runtime guard as two separate
  services with separate listeners, protocols, authentication, dispatch, and
  resource ownership.
- Governed workloads never receive the daemon control socket.
- Runtime guard endpoints remain isolated per session even if their execution
  capacity becomes shared.
- The daemon remains the privileged admission and lifecycle authority.
- `SessionSpec` remains immutable and pins the exact admitted runner
  capability, identities, policy inputs, failure behavior, output
  requirements, and workload privilege constraints.
- The durable session repository, lifecycle generation checks, idempotency,
  safe descriptor-based path resolution, required-output fail-closed behavior,
  root-only audited retention holds, and honest recovery states remain.
- A runner owns its execution and recovery mechanisms. Adding a runner must
  not require embedding that runner's process, container, or orchestration
  behavior in a generic core module.
- Linux is the first implementation, not a universal mechanism imposed on
  macOS, Docker, Kubernetes, or another future runner.
- No compatibility layer is required for the Phase 2 internal daemon-session
  protocol. A breaking cleanup is allowed only when explicitly approved.
- No suggestion may silently weaken containment, interception, output,
  evidence, or daemon-failure guarantees to reduce code.
- Do not implement this proposal until the user approves exact suggestions and
  the first implementation slice.

## Current Baseline

The reviewed commits add the following main path:

```text
erebor client
  |
  v
daemon control listener
  |
  v
DaemonControlState
  |
  v
DaemonSessionService
  |-- admission and start revalidation
  |-- staging mounts and guard credentials
  |-- output stores and input leases
  |-- terminal resource cleanup
  |
  v
SessionSupervisor
  |-- lifecycle transitions
  |-- active handle map
  |-- start, wait, stop, kill, remove, recovery
  |
  +--> SessionRepository
  |
  +--> RunnerRegistry
         |
         +--> LinuxHostSessionRunner
         +--> DockerSessionRunner
                |
                v
        generic erebor-session-helper
                |
                +--> LinuxWorkload
                +--> DockerWorkload

agent process guard
  |
  v
per-session runtime guard socket
  |
  v
per-session acceptor and worker pool
```

The current behavior is intentionally durable and defensive:

1. The daemon authenticates the Unix peer and decodes a typed request.
2. Mutating requests are made durable through the idempotency store.
3. Create admission resolves caller paths through the descriptor broker,
   snapshots runner capability and policy inputs, and persists `created`
   before starting resources.
4. Start revalidates the admission, transitions to `starting`, creates output
   stores, staging mounts, guard credentials, and a per-session guard service.
5. The selected runner starts a common helper, and the helper starts either a
   Linux process-guard workload or a Docker container.
6. The supervisor persists the stable runner identity and transitions to
   `running`.
7. The helper owns output capture and applies the selected behavior when the
   daemon control pipe reaches EOF.
8. After daemon restart, the supervisor reconstructs a Linux or Docker active
   handle from the durable runner binding rather than reconnecting to the old
   daemon pipe.
9. Terminal reduction stops the guard, removes leases and credentials,
   unmounts staging, and preserves or removes durable records according to the
   requested operation and retention hold.

## Main Simplification Finding

The security contracts are not the primary source of accidental complexity.
The largest issue is that one session lifecycle is coordinated by several
partial owners:

- `DaemonSessionService` prepares and cleans runtime resources;
- `SessionSupervisor` owns state reduction and active handles;
- `SessionRepository` owns durable state validation;
- the core crate owns Linux and Docker launch/recovery implementations; and
- the common helper again selects Linux or Docker workload behavior.

This divides one start or recovery transaction across multiple modules and
repeats runner selection at several layers. Failure cleanup must consequently
be manually coordinated across those owners.

## S1: One Complete Session Lifecycle Owner

### Recommendation

Approve.

### Current Behavior

`DaemonSessionService` at
`crates/erebor-runtime-daemon/src/session_control.rs:55` owns:

- conversion from admitted requests into session operations;
- start-time root-configuration revalidation;
- safe-path identity revalidation;
- output-store creation;
- staging mounts;
- runtime-guard credential creation and registration;
- input-lease managers;
- runtime cleanup; and
- protobuf response construction.

`SessionSupervisor` at
`crates/erebor-runtime-session/src/session_supervisor.rs:67` separately owns:

- runner discovery and capability comparison;
- lifecycle transitions;
- runner start and recovery;
- the in-memory active-handle map;
- wait, stop, kill, remove, and reconcile reduction; and
- terminal state selection from runner exit facts.

`SessionRepository` at
`crates/erebor-runtime-session/src/session_repository.rs:80` owns durable
generation comparison and validates the lifecycle edge again before writing.

A start currently crosses both `DaemonSessionService::start` at
`session_control.rs:402` and `SessionSupervisor::begin_start/launch_start` at
`session_supervisor.rs:136` and `session_supervisor.rs:164`. Each layer has
its own failure cleanup.

### Problem

There is no single owner of the start, recovery, or terminal-cleanup
transaction.

Examples:

- A staging failure must be converted into a lifecycle failure by one owner
  and cleaned by another.
- A runner-start failure must separately stop the guard and clean mounts.
- `wait`, `stop`, `kill`, and `remove` repeat terminal cleanup outside the
  supervisor that selected the terminal state.
- Repository-only operations such as inspect, list, prune, and retention hold
  pass through supervisor methods that add no domain behavior.

This makes crash points and compensation hard to enumerate.

### Proposed Target

Replace `SessionSupervisor` with one complete `SessionManager` in
`erebor-runtime-session`.

`SessionManager` owns:

- `SessionRepository`;
- `RunnerRegistry`;
- the active-session table;
- prepared runtime resources;
- runtime-guard registrations;
- output stores;
- start, recovery, terminal cleanup, remove, and prune;
- lifecycle transition decisions; and
- root-only retention state mutation as a domain operation.

The daemon-side owner becomes a thin `DaemonSessionApi` that:

- converts protobuf requests into domain commands;
- provides the observed caller identity and current root constraints;
- calls `SessionManager`; and
- converts domain results into protobuf responses.

Safe caller-path resolution remains a real privilege boundary. The manager
receives a narrow safe-path resolver collaborator; it does not open raw
caller paths itself.

Proposed ownership:

```text
crates/erebor-runtime-session/src/session_manager.rs
crates/erebor-runtime-session/src/session_manager/
  active.rs
  lifecycle.rs
  recovery.rs
  resources.rs

crates/erebor-runtime-daemon/src/session_api.rs
crates/erebor-runtime-daemon/src/session_api/
  admission.rs
  protocol.rs
  response.rs
```

The exact file split is secondary. The required architecture is one lifecycle
owner and one daemon protocol adapter.

### Behavior That Must Remain

- Create persists `created` before creating live resources.
- Start is legal exactly once unless an idempotency replay resumes the same
  pending operation.
- Runner capability and root constraints are revalidated immediately before
  start.
- No workload reaches `running` before the stable runner binding is durable.
- Every failure point produces an honest durable state.
- Terminal cleanup is idempotent and safe to retry after daemon restart.
- Retention hold continues to block removal and prune.

### Risks

- Moving root-resource preparation into the lifecycle transaction could
  accidentally make the session crate responsible for request authentication.
  It must not; the daemon remains responsible for peer authentication and
  authorization.
- A manager that directly depends on daemon configuration would recreate the
  same boundary under another name. Pass a validated root-constraint snapshot
  instead.
- One large manager file would not be a simplification. The owner may use
  focused collaborators, but only the manager commits lifecycle outcomes.

### Acceptance Evidence

- Fault injection at every start step proves one cleanup path and one final
  durable state.
- Replaying the same idempotent start across each crash point produces no
  duplicate helper, guard, mount, process, or container.
- Stop, kill, wait, natural exit, failed required output, remove, and recovery
  all invoke the same terminal finalization owner.
- Existing repository generation-race tests remain.

### Decision

- [ ] Approve
- [ ] Reject
- [ ] Revise

## S2: Explicit Prepared-Session Resource Owner

### Recommendation

Approve. This suggestion supports S1 but can be reviewed independently.

### Current Behavior

`OutputEndpoints` at
`crates/erebor-runtime-core/src/session.rs:43` begins as five output paths but
also accumulates:

- runtime environment variables;
- a prepared workspace path; and
- a prepared executable path.

`DaemonSessionService::prepare_runtime` at
`crates/erebor-runtime-daemon/src/session_control.rs:638` creates output
stores, staging mounts, guard credentials, a guard registration, and the
runner environment, then returns that `OutputEndpoints` value.

Cleanup is manual in:

- `DaemonSessionService::start`;
- `DaemonSessionService::finalize_runtime_resources`; and
- `DaemonSessionService::cleanup_staging`.

### Problem

`OutputEndpoints` is not an output-only value. It is an incomplete runtime
resource bundle without ownership or rollback semantics.

If preparation fails after mounting a workspace or starting a guard, callers
must know exactly which earlier steps succeeded. New resources added in later
phases would add more compensation branches.

Absolute output and runtime paths are also constructed in several locations
from UID, session ID, and configured roots.

### Proposed Target

Introduce two owners:

`SessionLayout`

- deterministically derives state, output, runtime, staging, credential, and
  socket paths from trusted roots, owner UID, and session ID;
- contains no IO and no user-controlled path resolution.

`PreparedSession`

- owns successfully created output stores, staging mounts, resolved execution
  paths, guard registration, runtime environment, and runner-specific prepared
  input;
- automatically rolls back owned resources if start fails before activation;
- transfers cleanup ownership to the active session entry only after the
  runner binding and `running` state are durable; and
- supports idempotent reconstruction during daemon recovery.

Proposed locations:

```text
crates/erebor-runtime-session/src/session_layout.rs
crates/erebor-runtime-session/src/session_manager/resources.rs
```

`OutputEndpoints` is removed. A runner receives an immutable
`PreparedSession` view containing only the resources admitted for that runner.

### Behavior That Must Remain

- Required stdout/stderr configuration continues to come from `OutputPlan`.
- Output stores exist before the runner starts.
- A user path is never reopened by pathname as root.
- Docker receives only daemon-staged paths.
- Linux receives only its private projected resources.
- Guard credentials are never included in public session responses or normal
  workload output.

### Risks

- Destructor-only cleanup cannot report errors. Explicit `rollback()` and
  `finalize()` operations must return cleanup failures; `Drop` is only the
  last-resort best effort.
- Recovery must verify existing staging resources rather than recreate or
  trust them blindly.
- A generic resource bag with many optional fields would repeat the existing
  problem. Runner-specific prepared data should remain runner-owned.

### Acceptance Evidence

- Tests fail preparation after each acquired resource and prove no leaked
  mount, guard socket, credential, output handle, process, or container.
- Repeated rollback and terminal cleanup are harmless.
- `SessionLayout` path tests cover UID/session separation and unsafe session
  identifiers.

### Decision

- [ ] Approve
- [ ] Reject
- [ ] Revise

## S3: Runner Implementations Leave `erebor-runtime-core`

### Recommendation

Approve.

### Current Behavior

The generic traits are correctly defined in
`crates/erebor-runtime-core/src/session.rs:191` and
`crates/erebor-runtime-core/src/session.rs:205`.

However, the same core crate also owns:

- `SessionHelperLaunchConfig` in
  `crates/erebor-runtime-core/src/session/active.rs:26`;
- helper process launching and the JSON helper protocol;
- Linux recovery in `active.rs:368`;
- Docker recovery in `active.rs:488`;
- Docker CLI inspection;
- process start-time parsing and process-group signaling;
- `LinuxHostSessionRunner`; and
- `DockerSessionRunner`.

Both runner structs also serve the old foreground CLI path and the new
daemon-owned active-session path.

### Problem

Adding a runner requires editing generic core matches and helper matches.
Linux/Docker process behavior causes `erebor-runtime-core` to depend on
`rustix`, and core knows about helper paths, Docker paths, systemd paths,
container inspection, and Linux process identity.

The `SessionRunner` seam therefore exists, but its implementations are still
centralized on the generic side of the seam.

### Proposed Target

Keep only generic session contracts in `erebor-runtime-core`:

```text
crates/erebor-runtime-core/src/session/
  spec.rs
  lifecycle.rs
  runner.rs
```

Move daemon-owned implementations to:

```text
crates/erebor-runtime-session/src/runners.rs
crates/erebor-runtime-session/src/runners/
  linux.rs
  linux/
    ...
  docker.rs
  docker/
    ...
```

Each runner owns:

- capability inspection;
- runner-specific admission validation;
- preparation requirements;
- start;
- observation;
- signal semantics;
- recovery proof;
- runner-resource cleanup; and
- encoding and validation of its durable recovery handle.

`RunnerRegistry` is the only component that selects a runner.

### Runner Identifier Decision

The current `SessionRunnerKind` enum requires central edits for every new
runner.

Recommended target: a validated `RunnerId` such as `linux-host`, `docker`,
`macos`, or `kubernetes`, used as the registry key and persisted in
`SessionSpec`.

Alternative: retain `SessionRunnerKind` while moving implementations. This is
less disruptive but preserves the central extension point.

Select one:

- [ ] Use validated extensible `RunnerId` throughout the daemon-owned path
- [ ] Retain the enum and only move implementations
- [ ] Revise

### Behavior That Must Remain

- The same selected runner performs inspect, validate, start, recover, signal,
  and cleanup.
- The exact capability snapshot remains durable.
- Start and recovery fail if the current implementation cannot satisfy the
  admitted snapshot.
- Linux and Docker retain their current containment and no-implicit-pull
  guarantees.

### Risks

- Maintaining separate foreground and daemon runner implementations for too
  long would create duplication. The foreground implementation is removed
  when its approved CLI migration phase lands.
- An opaque string runner ID still needs strict validation and collision
  rules.
- Runner registration must be explicit; runtime loading of arbitrary code
  into the privileged daemon is not introduced.

### Acceptance Evidence

- Core has no Docker CLI, systemd, helper-process, Linux process, or recovery
  implementation.
- A fake runner can be registered in tests without changing a central enum
  match.
- Linux and Docker capability/start/recovery tests live beside their runner
  owners.

### Decision

- [ ] Approve
- [ ] Reject
- [ ] Revise

## S4: Runner-Owned Continuity And Recovery Handle

### Recommendation

Approve the ownership rule. Review the exact Linux and Docker mechanisms
before implementation.

### Current Behavior

`SessionHelperLaunchConfig` in core starts one common
`erebor-session-helper`.

The helper receives a `SessionHelperHandoff` containing:

- the complete `SessionSpec`;
- output and continuity paths;
- prepared workspace/executable paths;
- runtime environment;
- process-guard and Docker executable paths; and
- systemd unit and slice names.

`HelperWorkload` at
`crates/erebor-runtime-session/src/session_helper/workload.rs:25` selects
`LinuxWorkload` or `DockerWorkload`.

The initial daemon controls the helper over inherited stdin/stdout. After
daemon loss, that pipe reaches EOF and cannot be reconnected. A restarted
daemon does not regain the helper protocol. Instead:

- `RecoveredLinuxSession` parses the saved process-group and helper PID from a
  semicolon-delimited string and signals the process group directly.
- `RecoveredDockerSession` parses the container/helper identities, inspects
  Docker directly, and controls the container directly.
- Both use helper liveness as part of recovery proof, while durable continuity
  evidence is not the sole recovery authority.

The complete Linux `continue` acceptance currently fails because the saved
process identity is no longer live after daemon SIGKILL.

### Problem

The common helper is a Linux implementation mechanism presented as a generic
runner abstraction.

Runner choice appears in:

- the registry;
- core runner implementations;
- helper workload selection; and
- core recovery implementations.

The saved `RunnerBinding::stable_identity` is a string with runner-specific
format and delimiters. Generic code parses runner-specific recovery facts.

### Proposed Target

Make continuity part of the runner contract:

```text
Runner
  inspect()
  validate_admission()
  prepare()
  start() -> ActiveSession + RecoveryHandle
  recover(RecoveryHandle) -> RecoveryObservation
  signal()
  cleanup()
```

`RecoveryHandle` is:

- versioned;
- opaque outside the selected runner;
- durably stored with runner ID and implementation ID;
- bounded in size;
- validated by the runner before use; and
- never interpreted by generic core or daemon protocol code.

Each runner chooses mechanisms suitable to its execution environment. A
runner may use an operating-system service manager, cgroup identity, a
runner-specific continuity process, a container runtime, or an orchestrator.
No one mechanism becomes a requirement for every runner.

Shared internal libraries are allowed where they remove duplication, but a
shared library does not own runner selection.

### Mechanism Decisions Required

Linux-host:

- [ ] Keep a Linux-specific continuity helper and define durable recovery
      evidence that a restarted daemon can verify
- [ ] Use the Linux service/cgroup manager as the primary durable workload
      identity and retain only the minimum helper needed for output or loss
      behavior
- [ ] Revise

Docker:

- [ ] Use Docker container identity as the primary recovery handle and retain
      a runner-owned loss watcher only where required
- [ ] Keep a Docker-specific continuity helper with explicit durable recovery
      evidence
- [ ] Revise

The selected mechanisms must separately prove `terminate` and `continue`.

### Behavior That Must Remain

- `terminate` kills the complete admitted workload after daemon loss.
- `continue` preserves the workload, output, and sufficient recovery evidence.
- A restarted daemon returns to `running` only after proving runner identity,
  containment, guard/evidence continuity, and output continuity.
- A missing or mismatched recovery fact results in an honest terminal or
  interrupted state.
- Recovery never trusts a PID by itself.

### Risks

- Removing the common helper without replacing its daemon-loss observation
  would break `terminate`.
- Treating a container ID, PID, or unit name as sufficient identity would
  weaken recovery.
- A reconnectable helper endpoint would become another authenticated service
  and requires an explicit security design; it must not be introduced
  implicitly.
- Runner-specific mechanisms can share contract tests, but their internals
  should not be forced into identical shapes.

### Acceptance Evidence

- Direct daemon SIGKILL and graceful service restart pass for every advertised
  runner/failure-mode pair.
- Recovery-handle corruption, reuse, stale identity, missing helper/watcher,
  and changed containment all fail honestly.
- The Linux `continue` scenario returns to `running` only when continuity is
  proven.
- The complete helper/process/container tree is absent after `terminate`.

### Decision

- [ ] Approve the runner-owned continuity rule
- [ ] Reject
- [ ] Revise

## S5: One Daemon-Wide Runtime-Guard Capacity Owner

### Recommendation

Approve.

### Current Behavior

`RuntimeGuardService` at
`crates/erebor-runtime-session/src/runtime_interception_broker/service.rs:23`
stores default limits and creates one
`RuntimeInterceptionBrokerServer` per session.

Each Unix server at
`crates/erebor-runtime-session/src/runtime_interception_broker/platform.rs:175`
creates:

- one acceptor thread;
- a fixed worker pool, currently four workers by default;
- one bounded queue;
- one connection budget; and
- one shutdown/drain lifecycle.

The latest hardening commit correctly removes unbounded thread-per-connection
behavior, but the bounds apply to each session independently. There is no
daemon-wide active-session or guard-thread budget.

At the current defaults, 100 active guard sessions can create approximately
500 guard threads before counting helper and output-pump threads.

### Problem

`RuntimeGuardService` is the apparent capacity owner, but it multiplies the
same capacity for every session. Total daemon resource use remains
proportional to an otherwise unbounded active-session count.

### Proposed Target

Keep one isolated socket and registration per session, but give the service:

- one shared bounded guard executor;
- one global connection budget;
- one global queue;
- smaller per-session connection and queued-request limits for fairness;
- one deadline/cancellation policy;
- service-wide shutdown and drain; and
- per-session unregister/drain without stopping unrelated sessions.

Practical target ownership:

```text
runtime_interception_broker/
  service.rs       # session registration and service lifecycle
  executor.rs      # shared queue, workers, global/per-session permits
  platform.rs      # Unix listener/socket operations only
  server.rs        # authenticated protocol dispatch only
```

The implementation may retain one lightweight accept source per socket or use
a platform event mechanism. The architectural requirement is that worker and
connection capacity are shared and globally bounded.

### Behavior That Must Remain

- Daemon control and runtime guard remain separate services.
- Session A cannot authenticate to or dispatch through session B's socket.
- Per-session sockets retain their existing filesystem isolation.
- Overload fails closed before domain dispatch.
- Read and write deadlines remain.
- Stopping one session drains only that session's queued/active work.
- Stopping the service drains all work within a bounded deadline.

### Risks

- A global queue without per-session fairness lets one session starve others.
- Holding global locks during policy evaluation could serialize independent
  sessions.
- Reusing the daemon control executor would merge service resource ownership;
  the runtime guard keeps its own pool.

### Acceptance Evidence

- Starting many sessions does not create workers per session.
- Global connection/queue limits remain fixed under multi-session load.
- Per-session overload does not block a second session.
- Session unregister cancels or drains only its own work.
- Service shutdown has a deterministic deadline and leaves no sockets or
  workers.

### Decision

- [ ] Approve
- [ ] Reject
- [ ] Revise

## S6: Compose `SessionSpec` From Smaller Validated Sections

### Recommendation

Approve.

### Current Behavior

`RunRequest` at
`crates/erebor-runtime-core/src/session/admission.rs:714` is a large flat input
value.

`SessionAdmission` at `admission.rs:883` then mirrors almost every field that
will appear in `SessionSpec`.

`SessionSpec` at `admission.rs:912` copies those fields again and owns one long
validation method that mixes:

- generic identity validation;
- runner selection;
- Linux versus Docker execution requirements;
- policy identity completeness;
- capability compatibility;
- path validation;
- output validation;
- TTY/attach assumptions; and
- lifecycle timestamps and configuration generation.

`OutputPlan` also stores an absolute daemon state path even though that path is
deterministically derived from trusted daemon roots, UID, and session ID.

### Problem

The immutable contract is valuable, but its flat representation makes all
changes look like changes to one object. Runner-specific validation lives in
generic spec validation, and construction requires a field-for-field mirror
type.

This will become harder when macOS, Kubernetes, packages, adapters, secrets,
and new governed surfaces are added.

### Proposed Target

Compose the spec from validated domain sections:

```text
SessionSpec
  identity: SessionIdentity
  execution: ExecutionPlan
  governance: GovernancePlan
  continuity: ContinuityPlan
  output: OutputPlan
  admission: AdmissionStamp
```

Suggested responsibilities:

`SessionIdentity`

- session ID;
- owner UID/GID;
- creation time.

`ExecutionPlan`

- runner ID;
- command;
- immutable workspace/executable/image identities;
- environment;
- workload privilege constraints;
- runner-owned admitted execution data.

`GovernancePlan`

- package/installation/adapter identities;
- policy-input identities and effective policy-set digest;
- admitted governed surfaces and evidence requirements.

`ContinuityPlan`

- daemon-failure mode;
- root-capped loss grace;
- required recovery/output continuity.

`OutputPlan`

- maximum bytes;
- rotation;
- page size;
- required/optional streams and evidence.

`AdmissionStamp`

- spec schema version;
- root configuration generation;
- complete runner capability snapshot.

`SessionAdmission` is removed. The admission owner constructs these validated
sections and then constructs `SessionSpec`.

Physical state/runtime/output paths move to `SessionLayout`; immutable limits
and requirements remain in `SessionSpec`.

Generic `SessionSpec` validation checks composition and cross-section
invariants. Runner-specific executable/image/containment validation belongs to
the selected runner.

### Behavior That Must Remain

- The complete effective admission remains serializable and durable.
- Deserialization revalidates every nested section.
- Policy/package/runner identities remain cryptographically pinned.
- The exact runner capability snapshot remains part of the immutable record.
- Runtime paths remain daemon-derived and cannot become user-selected.

### Risks

- Splitting one flat object into many empty wrappers would add ceremony rather
  than simplify ownership. Each section must own real validation and behavior.
- Runner-specific admitted data must remain versioned and bounded.
- Moving absolute paths out of the spec must not make recovery depend on
  ambient current configuration; `SessionLayout` uses the durable session
  owner/ID and the installed daemon roots.

### Acceptance Evidence

- No field-for-field `SessionAdmission` mirror remains.
- Generic spec validation contains no Linux/Docker match.
- Round-trip and malformed nested-section tests cover every section.
- Existing records are intentionally migrated or rejected according to the
  approved no-compatibility decision; no silent reinterpretation occurs.

### Decision

- [ ] Approve
- [ ] Reject
- [ ] Revise

## S8: One Typed Daemon-Command Dispatch Model

### Recommendation

Defer until S1-S7 are decided, then approve if the remaining command set still
has the same duplication.

### Current Behavior

`DaemonControlState::dispatch` at
`crates/erebor-runtime-daemon/src/control.rs:362` matches message-kind strings
to many request-specific methods.

Mutating session methods then:

1. decode a protobuf message;
2. derive UID/session scope;
3. choose an operation string;
4. build a `MutationIntent`;
5. invoke generic idempotency;
6. match the intent again in `DaemonSessionService::apply`; and
7. encode a response kind/payload pair.

Root administrative commands repeat another authorization and audit path.
The client in `crates/erebor-runtime-client/src/session.rs` mirrors the command
and response-kind mapping.

### Problem

Message kind, mutability, authorization scope, operation name, intent, and
response kind are related facts represented in different match statements and
string constants.

Adding a command can compile while one metadata path is omitted.

### Proposed Target

Keep typed protobuf requests. Introduce one explicit typed daemon command
model:

```text
DaemonCommand
  decode(envelope)
  authorization_scope()
  idempotency_operation()
  execute(context)
  response_kind()
```

The model may be an enum with methods or a small explicit command registry.
Avoid a macro or generated framework that hides authorization and mutation
behavior.

Separate command families remain:

- daemon administration;
- user session commands; and
- root session administration.

Runtime guard and hook traffic do not enter this command model.

### Behavior That Must Remain

- Header validation occurs before domain dispatch.
- Unix peer identity remains authoritative.
- Root-only commands remain wire-distinct and audited.
- Exact received protobuf bytes remain part of the idempotency fingerprint.
- Read-only requests reject the daemon idempotency header.
- Mutating requests require it.

### Risks

- A generic command framework can obscure authorization.
- Combining daemon-control and runtime-guard messages would violate the
  service boundary.
- Response streaming for logs/events should remain explicit rather than forced
  into a unary abstraction.

### Acceptance Evidence

- One inventory test covers every accepted message kind and its mutability,
  authorization scope, operation identity, and response family.
- Cross-service and header-rejection tests remain.
- Root-only command audit tests remain.
