# Phase 2: SessionSpec, Active Runners, And Daemon-Failure Contract

Status: Done — the durable-session, open runner-driver boundary, separate
Linux/Docker continuity controllers, shared runtime-guard service, complete
installed daemon-loss matrix, and repository quality gate pass as of
2026-07-20. See the Phase 2 Result for exact evidence.

Companion review proposal:
[Phase 2 Architectural Simplification](phase-2-architectural-simplification.md).
It has no pending Phase 2 simplification.

## Purpose

Move durable session admission and active workload ownership behind `erebord`
while retaining the old public session CLI until the new daemon path is proven
internally.

The phase must make daemon death behavior a real runner/guard capability, not a
comment in configuration.

## Scope

### Immutable Admission

- Add validated core owners for `RunRequest`, `SessionSpec`,
  `RunnerBinding`, `DaemonFailureMode`, runner capabilities, filesystem
  projection, output plan, safe path bindings, admitted environment/secret
  references, and admitted surface/evidence requirements.
- Define the versioned `RunnerCapabilityDocument` now. It contains the runner
  implementation id/version, schema version, observed daemon-host OS and
  architecture constraints, filesystem and interception guarantees,
  stream/TTY/signal features, supported daemon-failure modes, and
  runner-specific admission constraints. It describes the effective runner
  implementation on this host, not portable guarantees attached to a runner
  name. Linux-host and Docker each produce a validated document. Phase 6 may
  extend it only additively; it does not redefine the contract.
- `SessionSpec` must contain at least:
  - cryptographically strong daemon-generated session id and observed owner
    UID/GID;
  - generic command or resolved package/installation/adapter identities;
  - immutable policy input revisions and resolved composition digest;
  - runner id, implementation id/version, and exact serialized
    `RunnerCapabilityDocument` snapshot;
  - safely resolved workspace and approved executable or Docker image identity;
  - sanitized environment variable names/non-secret values and secret-provider
    references, never resolved secret values;
  - private filesystem and endpoint projection plan;
  - bounded output/rotation, lifecycle, audit, and evidence paths;
  - resolved daemon-failure mode and root-capped loss grace; and
  - creation time and storage schema version.
- The daemon persists the complete validated spec and `created` state before
  preparing any side resource or runner.
- Until Phase 3 supplies real stores, internal e2e fixtures provide already
  validated immutable package, installation, and policy identities. Phase 2
  must not add a public unresolved/raw-config bypass for this test seam.
- Unresolved package, policy, runner, filesystem, endpoint, output, privilege,
  or failure-mode facts fail admission. `SessionSpec` is internal and must not
  become a user-facing profile file or CLI object.
- `create` binds immutable identities and creates durable records only. It does
  not create live namespaces, sockets, helpers, processes, or containers.
  `start` revalidates revocation, install/workspace/executable/image identity,
  policy, quotas, and current root constraints before entering `starting`.
- Record the effective root-configuration generation in the spec. Normal
  reload affects future create/start admission, not an active runner's pinned
  mode or filesystem plan. A live-session intervention must use an explicit
  audited admin/revocation operation, not mutate behavior invisibly through
  reload.

### Typed Session And Administration Protocol

- Extend Phase 1 `daemon.proto` with typed create/start/list/inspect/log/event/
  attach/stop/kill/wait/remove/prune messages and bounded stream cursors.
  Phase 2's e2e driver uses exactly these messages; no internal test-only
  control path may bypass daemon authentication or authorization.
- Require the Phase 1 daemon-only `erebor-idempotency-key` header on every
  mutating daemon-control request and reject it on read-only, guard, and hook
  traffic. After family/message validation, compute the request fingerprint as
  SHA-256 over the versioned domain separator, protocol major, daemon message
  kind, and exact received protobuf payload bytes, excluding envelope
  ids/correlation and all headers. The client supplies no digest.
- Before the side effect, persist scope `(observed UID, operation kind,
  idempotency key)`, the request fingerprint, and mutation state. The same
  key/fingerprint resumes or returns the original mutation/result; the same
  key with a different fingerprint rejects. A transport message id alone never
  proves retry safety.
- Once durably accepted, create/start/stop/kill/remove continues despite client
  timeout or disconnect. Retrying the same key or inspecting the object obtains
  the result; transport cancellation does not roll back or silently signal the
  workload.
- Retain each session-mutation idempotency record through at least the session
  tombstone/advertised retry horizon and never permit its key to be rebound
  while retained. GC rules and the response expose when safe retry guarantees
  expire.
- Add wire-distinct root-only administration messages for list/inspect/stop/
  kill across user namespaces. Only observed UID 0 may include a target UID;
  normal requests never do. Administrative actions are audited and do not
  include interactive attach or workload payload access by default.

### Durable Lifecycle Store

- Retire the filesystem I/O in
  `crates/erebor-runtime-core/src/session_registry.rs`. Keep pure validated
  records/transitions in core, but put the root-owned durable repository in
  `erebor-runtime-session` behind the daemon. It is UID-scoped and rooted at
  `/var/lib/erebor/users/<uid>/sessions/<session-id>/`.
- Implement every state edge from the parent README, including
  `created -> removed`, fast terminal exits directly from `starting`,
  stop/kill/natural-exit races, `control_lost` recovery or verified terminal
  reduction, and terminal removal. `rm --force` is an
  idempotent stop/kill/finalize/remove workflow, not a direct live-to-removed
  mutation. Retention/legal holds reject removal; accepted removal persists an
  audited tombstone before later GC of releasable bytes.
- Every transition validates its predecessor and durable generation. One owner
  serializes mutation; state writes use safe file creation plus
  write/fsync/rename and directory fsync before acknowledgement.
- A root-only, idempotent administrative retention-hold operation may place or
  release a hold for an exact `(uid, session-id)`. It returns the durable record
  and writes one daemon audit record when the requested value changes. A normal
  session owner cannot place a legal hold or remove a held session.
- Persist intent before side effects and result after observation. A crash may
  leave `starting`, `stopping`, or `control_lost`; it must not manufacture a
  terminal success. Persist the observed `RunnerBinding` after launch and
  before acknowledging `running`; it is mutable lifecycle state beside the
  immutable spec, not a field guessed during create.
- Keep daemon telemetry, workload output, lifecycle events, and governance
  evidence as distinct files/streams.

### Active Runner Contract

- Put daemon-owned execution behind one `RunnerDriver` extension boundary in
  `erebor-runtime-session/src/runners.rs`:

  ```text
  RunnerDriver.inspect() -> RunnerCapabilityDocument
  RunnerDriver.start(SessionSpec, OutputEndpoints) -> ActiveSession

  ActiveSession:
    recovery() -> opaque versioned RunnerRecovery
    capability_snapshot()
    wait()
    stop(grace_period)
    kill(signal)
    health()
    attach/read streams when supported
  ```

- The runner registry calls `inspect` before admission. The daemon validates
  and snapshots that exact capability document into `SessionSpec`; `start`
  revalidates that the same implementation remains available and compatible
  with the admitted snapshot before it creates a workload. Capability
  discovery cannot be deferred until an `ActiveSession` already exists.
- Move both current implementations in
  `crates/erebor-runtime-core/src/session/linux_host.rs` and `docker.rs` to the
  contract before the phase is accepted. A blocking `Command::status` or
  `Command::output` is not an active session.
- Add one compiled runner registry that maps an immutable runner id to a
  `SessionRunner` implementation and returns unavailable with a stable reason
  when that implementation cannot run on the current daemon host. This
  registry is the only runner-selection switch; the generic session
  supervisor and daemon dispatchers must not branch on Linux-host, Docker, or
  future macOS behavior.
- Linux-host owns a daemon-created cgroup/process-tree identity, root pidfd or
  equivalent start-time-qualified identity, process group, guard, pipe
  streams, and complete-tree termination. A process group alone is not
  accepted because a child can create another session/group. The active
  contract can represent a PTY, but the initial Phase 2 Linux-host and Docker
  capability documents both report `tty_supported=false`; interactive PTY and
  stdin transport are Phase 6 parity work.
  Physical process interception reports unavailable on architectures for which
  the standalone guard does not compile (currently anything outside the
  explicitly supported Linux target matrix) instead of advertising partial
  enforcement.
- Docker owns the exact container id, inspect/wait/log/attach handles, and
  termination. Its image is an exact already-local digest/image id and start
  disables implicit pull; a mutable tag cannot reach this contract. Do not
  reconcile by container name or a bare PID.
- Put Docker operations behind one runner-owned collaborator now. The initial
  implementation may use the existing Docker CLI only as an explicit,
  version-probed integration boundary with structured output parsing, bounded
  subprocesses, clean environment, cancellation, and typed errors; command
  strings do not leak into the session supervisor.
- Each runner owns execution-specific workload creation, containment, stable
  identity, guard preparation, endpoint projection, streams/TTY,
  stop/kill/wait, recovery observation, and its declared daemon-loss behavior
  behind `ActiveSession`. The session supervisor in
  `erebor-runtime-session` owns the active handle, generic policy/surface
  coordination, output/evidence contracts, durable state transitions, and
  final reduction. Neither the supervisor nor the daemon dispatch layer may
  switch on runner internals.

### Runtime Guard Service Boundary

- Refactor the static/singleton server side in
  `crates/erebor-runtime-session/src/runtime_interception_broker.rs` into one
  explicitly constructible session-domain server with injected endpoint,
  authentication, and registration owners. The session crate must not depend
  on the daemon crate.
- Host that domain server behind a named runtime guard service inside the
  existing `erebord` process for daemon-owned sessions. A temporary foreground
  adapter constructs the same server for the direct-execution path inside
  `erebor`. This is a second
  service boundary in the same daemon process, not duplicated enforcement, a
  second daemon executable, systemd unit, or competing control plane. Phase 3
  removes the generic foreground adapter; Phase 4 removes its final Codex use.
- The Phase 1 daemon control service continues to own only
  `/run/erebor/daemon.sock` and `Daemon*` messages. The runtime guard service
  owns one shared host listener at
  `/run/erebor/sessions/runtime-interception.sock`, logical UID/session
  registrations, and only `Guard*`, interception, and guard-lifecycle
  messages. Each workload sees that socket only through a private session-local
  projection and receives only its own credential. The guard service must not
  reuse the daemon control listener, peer-auth object, connection state
  machine, dispatcher, or request queue. Each service runtime owns its own
  connection limits, queues, deadlines, cancellation, dispatch, draining, and
  shutdown; shared IPC codecs own only bounded frame I/O.
- The standalone Linux process guard is a client of the runtime guard service:
  it connects to its exact admitted session endpoint, authenticates with a
  short-lived session credential plus independently observed process evidence,
  sends `GuardHello`, and then sends physical-effect/lifecycle facts. The
  service binds the connection to one immutable session before dispatch.
  Caller-selected UID, session ownership, policy, or authorization fields are
  never trusted from a guard payload.
- The runtime guard service routes a bound fact through the session's narrow
  interception/lifecycle owner. It does not resolve packages, accept CLI
  commands, perform daemon administration, or expose another user's
  registration. A daemon-control handler cannot forward an arbitrary envelope
  into it.
- `erebord` owns creation, permissions, registration, draining, and removal of
  each guard socket as part of the durable session lifecycle. `create` opens no
  listener. `start` persists intent before creating the listener; terminal
  finalization closes it and removes only the exact session artifact. Restart
  reconciles listener state from the durable session and stable runner/guard
  identity rather than trusting a pathname.
- Keep Codex hook endpoints separate from both primary services. This phase
  preserves the real Codex hook broker in the temporary direct foreground Codex
  path inside `erebor` and
  documents its final boundary only; it must not add a production daemon hook
  listener or placeholder. Phase 4 moves that real adapter-owned broker into
  `erebord` as a session resource with its own hook message family, ticket,
  peer-evidence, queues, and lifecycle. It is never daemon-control or
  runtime-guard traffic and is not a separate host daemon.
- Linux x86_64 is the first real guard client. The service contract may model
  a future macOS guard, but this phase must report macOS unavailable and must
  not claim native interception until a separate native-runner plan and
  implementation prove it. Docker advertises guard/interception capabilities
  only when its runner has a real admitted implementation.

### Privilege And Filesystem Boundary

- The root daemon performs only privileged preparation, then starts the
  workload as the peer-observed UID/GID with supplementary groups, umask,
  environment, cwd, and resource limits resolved by admission. It calls
  `env_clear` (or equivalent) and builds an explicit safe environment; root
  daemon variables are never inherited by default.
- Close all unadmitted file descriptors with a close-range/CLOEXEC allowlist
  before exec. The child must not inherit the daemon listener, configuration,
  registry credential, trust-store descriptor, another session's endpoint, or
  a privileged namespace handle.
- Resolve caller-provided workspaces, executables, installations, and import
  paths through a short-lived root-owned descriptor broker. Before resolving,
  it clears supplementary groups, permanently drops to the caller's GID/UID,
  closes unrelated descriptors, and has no network access. It performs
  descriptor-relative `openat2`/no-follow resolution and returns held
  descriptors with `SCM_RIGHTS` plus `statx` device/inode/mount/owner identity
  over a private socketpair. `erebord` hashes, copies, mounts, or `execveat`s
  from those descriptors and never reopens the raw request string. Small
  bounded uploads may use the control upload transaction; live trees,
  executables, and OCI layouts use this broker.
- Linux-host creates a private mount namespace and mounts a private
  `/run/erebor` during start. Session-owned host endpoints live below the
  root-owned, no-follow `/run/erebor/sessions/<uid>/<session-id>/`; the shared
  guard listener lives at `/run/erebor/sessions/runtime-interception.sock`.
  The workload's `/run/erebor/daemon.sock` and other users' state are absent.
  Only exact admitted guard, hook, and surface endpoints are bind-projected.
  The guard projection reaches the runtime guard service, never the daemon
  control service.
- Docker never mounts the host daemon socket or another user's state. It
  projects only declared artifacts/endpoints and records their container paths
  in `SessionSpec`. Its container process uses the admitted UID/GID (or an
  explicitly root-authorized identity mapping); running as container root is
  not an implicit consequence of the host daemon being root.
- Docker bind mounts use daemon-created stable staging mounts below the
  session runtime root, derived from held/safely resolved descriptors. Never
  pass a user-controlled workspace/artifact pathname to Docker for it to
  resolve later as root.
- Read-only permissions do not substitute for socket absence. Tests must try
  both path lookup and connection from inside the governed workload, including
  alternate `/proc/*/root` and `/proc/*/fd` paths. The Linux runner must prevent
  mount/setns/inherited-descriptor escape from the admitted filesystem view or
  reject the required capability on that host.

### Daemon Control Lease And Failure Modes

The helper, cgroup, and systemd topology in this section is the Linux V1
implementation for the initial Linux-host and Docker runners. It is not part of
the cross-platform `ActiveSession` contract. A future runner must meet and
attest the same declared lifecycle outcomes using mechanisms it owns on its
supported daemon host.

- Give each initial runner its own private controller contract. Linux-host
  builds `/usr/libexec/erebor/erebor-linux-session-controller`; Docker builds
  `/usr/libexec/erebor/erebor-docker-session-controller`. Their handoff,
  command, event, and recovery types are distinct and contain no runner-kind
  dispatch. The generic supervisor launches neither directly.
- A controller has no listener and accepts no later client. Its entire
  authority is its runner-specific immutable startup handoff and inherited
  control-lease reader, output handles, and continuity handles. The lease
  writer is owned only by `erebord`, so EOF is unambiguous daemon loss without
  exposing the hidden daemon socket.
- Linux requires its controller to own the process guard and complete process
  tree. Docker retains its separate controller because the current Phase 2
  capability promises required output continuity and both `terminate` and
  `continue`; Docker's detached runtime alone cannot observe daemon-control
  EOF. A future Docker implementation may remove its controller only with a
  replacement continuity proof. Neither controller mutates authoritative
  session state.
- Create one independent systemd slice/cgroup subtree per session. Put the
  runner-specific controller in its own transient scope and the Linux workload
  or Docker container in its own sibling scope beneath that subtree. The slice
  and child scopes
  are not descendants of `erebord.service` and have no `PartOf=` or
  `BindsTo=` relationship to it. The daemon unit contains only `erebord` and
  private non-session workers.
- Treat direct daemon SIGKILL and `systemctl stop/restart erebord` as daemon
  loss, so the runner-specific controller applies the admitted mode.
  `erebor daemon stop`
  is the graceful RPC and refuses active unresolved sessions. Host shutdown is
  outside the initial `continue` guarantee. Record the exact unit/scope
  `KillMode`, delegation, restart, and cleanup behavior.
- Resolve a requested/package/policy/host mode against a per-runner capability
  report. Unsupported combinations fail admission:
  - `terminate`: both Linux-host and Docker must terminate the complete
    workload tree/container after control-lease loss.
  - `continue`: both Linux-host and Docker keep the workload running while the
    independent runner controller preserves output, lifecycle facts, and
    driver-owned recovery identity for later honest reduction.
  - `continue_if_enforced`: both initial runners reject it explicitly. A future
    runner guard may advertise it only when it independently enforces the
    complete exact pinned policy and proves required output/evidence continuity
    without any daemon-resident decision owner.
- On restart, reconcile from durable state and the driver's opaque recovery
  value.
  Continued work returns to `running` only after runner, guard, enforcement,
  output, and evidence continuity are verified; otherwise transition honestly
  toward termination or `interrupted`.
- A graceful daemon stop refuses while sessions are active unless the root
  operator explicitly resolves each one. The configured loss grace is
  root-capped so a user/package cannot make daemon loss effectively invisible.

### Output And Attach Foundations

- Capture stdout/stderr through the private helper into bounded session streams
  with sequence, timestamp, source, byte limit, and rotation/retention
  metadata. The daemon normally reduces and serves those streams; the helper
  continues appending during an admitted daemon-loss gap. A PTY has one
  terminal-output stream; input is not logged. For separate pipes the sequence
  is helper observation order, not a false claim about the workload's
  cross-descriptor write order.
- Advertise a resumable cursor only after the corresponding chunk/event is
  durably appended. A low-latency uncommitted observation, if ever exposed,
  must be labeled non-durable and cannot masquerade as a resume point.
- Implement the durable renewable input-lease owner and typed lease protocol,
  but issue no lease unless the active runner advertises an admitted PTY.
  Phase 2's two active runners report no PTY, so attach is read-only and an
  input-lease request rejects. Phase 6 owns real PTY/stdin transport and runner
  parity. Backpressure must not block the governed process indefinitely; the
  durable sink owns flow independently of a CLI connection.
- Client disconnect never terminates a detached session. Attached foreground
  behavior follows the admitted detach/TTY contract, not transport accident.
- A required output/evidence sink failure terminates or fails the session
  closed according to the admitted evidence contract. ENOSPC cannot silently
  convert a governed session into an unrecorded one.
- `OutputPlan` records whether stdout and stderr are required independently.
  A required stream read or durable-append failure reaches the helper,
  terminates the workload, and records a failed lifecycle result; an optional
  stream failure does not invent a failed completion merely because optional
  capture stopped.
- Stop/kill accept only runner-supported signal semantics and target the stored
  active handle. No daemon request accepts an arbitrary PID.

## Non-Goals

- Do not daemonize the existing direct governed-command implementations in
  this phase. They remain under the sole public `erebor` CLI until their
  approved Phase 3 or Phase 4 migration.
- Do not add public package/install/policy stores, OCI registry access, Codex
  package migration, Hub discovery, or Claude support.
- Do not move the real Codex hook broker into `erebord` or create a daemon hook
  placeholder; that migration belongs to Phase 4.
- Do not pretend all three failure modes are supported by every runner.

   daemon-loss behavior, restart reduction, and the full cross-runner lifecycle
   matrix.

may not expose a partially daemon-owned public session command.

## Acceptance

- `erebord` can internally admit, persist, start, observe, stop, kill, and
  recover a generic session through either current runner.
- The one `erebord` process hosts daemon control and runtime guard as separate
  service owners. They have separate listeners and dispatch paths even though
  both consume `erebor-runtime-ipc`.
- The real Linux process guard connects only to its private projected runtime
  guard endpoint with its session credential; neither guard nor workload can
  use the daemon control endpoint.
- No runner starts from unresolved mutable configuration.
- No governed workload can see or connect to the daemon control socket.
- Daemon death produces the admitted behavior or admission is refused.
- Linux-host and Docker support exactly `terminate` and `continue` in this
  phase and reject `continue_if_enforced`.
- Each current runner's private controller and independent session-owned
  systemd slice/cgroup subtree preserve the admitted session behavior without
  becoming another service or state owner. The controller protocols are
  separate and contain no runner-kind dispatch.
- Restart never reconstructs a running session from a PID alone.
- The existing public CLI has not yet been cut over.
