# Phase 5: Copy-Up Mutation And Metadata Semantics

Status: Proposed. Blocked on Phase 4 and explicit implementation approval.

## Purpose

Make the merged volume writable while guaranteeing that no session mutation
reaches the frozen lower or authoritative host workspace.

## Scope

- Implement per-object mutation serialization and crash-safe namespace journal
  ordering.
- On first mutation of a lower regular file, create an APFS clone in upper,
  verify it, preserve identity metadata, publish it atomically, and then apply
  the requested mutation.
- Implement create, exclusive create, open flags, read/write, append, truncate,
  allocate, hole/sparse behavior, fsync, and close semantics.
- Implement mkdir, rmdir, unlink, symlink, hardlink, and the directory-rename
  model approved after Phase 4.
- Implement file rename and supported macOS rename flags with atomic visibility
  matching the stable FSKit contract.
- Represent lower deletion with explicit metadata whiteouts; never create Linux
  whiteout devices or overlay xattrs.
- Preserve hardlink topology within upper and across copy-up. A mutation through
  one visible hardlink must be visible through every session-visible link to
  that object without mutating lower.
- Implement chmod, chown where allowed, timestamps, BSD flags, ACLs, xattrs,
  Finder metadata, and resource forks for supported objects.
- Define mmap and kernel-cache coherency behavior. Writable mappings must be
  accounted for by the freeze barrier and must never map the lower backing file
  writable.
- Enforce quotas/free-space failures without partially publishing a copy-up.
- Reject sockets, device nodes, FIFOs, named streams, unsupported clone flags,
  or metadata classes unless their exact semantics are implemented and tested.
- Persist enough journal state to replay or roll back an interrupted copy-up,
  rename, whiteout, or metadata update after extension restart.

## Mutation Ordering Contract

For a lower-backed mutation:

```text
reserve stable object identity
  -> journal copy-up intent
  -> clone/copy metadata into private temporary upper object
  -> fsync object and parent as required
  -> atomically publish upper mapping
  -> journal mapping committed
  -> apply requested mutation
  -> journal mutation committed
  -> acknowledge FSKit request
```

An acknowledged request must survive the documented synchronize/remount
boundary. An unacknowledged interrupted request may resolve to before or after
only where POSIX permits that outcome; it must never expose corrupt internal
state.

## Tests

- Native operation tests cover the full flag and metadata matrix.
- Concurrency tests race two first-writers, rename with open descriptors,
  unlink with open descriptors, hardlink mutation, readdir with rename, and
  xattr/ACL changes.
- Fault injection kills the extension or host at every journal boundary and
  verifies deterministic recovery.
- Real mount tests run shell redirection, `cp`, `mv`, `rm`, `ln`, `chmod`,
  `xattr`, `git checkout`, editor atomic-save patterns, mmap writes, and sparse
  file operations.
- Every scenario proves the authoritative host and frozen lower remain
  unchanged while upper/meta contain the session effect.

## Acceptance

- Supported POSIX and macOS mutations behave through a real FSKit mount.
- Lower and authoritative host content remain unchanged.
- Copy-up, rename, link, mmap, and metadata behavior is race-safe and
  recoverable.
- Unsupported objects fail before partial mutation and are auditable.
